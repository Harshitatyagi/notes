Design Pattern 3- Structural Design Pattern

What is a Structural Design Pattern?

A Structural Design Pattern focuses on the composition of classes and objects to form larger structures while keeping the system flexible and efficient. These patterns simplify the design by identifying simple ways to realize complex relationships between entities.

The primary goal of structural design patterns is to ease the design by ensuring that objects and classes can work together without tightly coupling them. This makes the system more scalable and maintainable.

Adapter Pattern

Scenario: Legacy Payment Gateway Integration

Imagine you are developing an e-commerce application. Your application integrates with a modern payment gateway that uses a standard interface to process payments. However, your company also has an older payment system (a legacy payment gateway) that needs to be supported, but its interface is completely different.

Problem Without Adapter Pattern

The modern payment gateway has a simple interface like this:

// Modern Payment Gateway Interface
interface ModernPaymentGateway {
    void processPayment(String accountNumber, double amount);
}

The legacy payment system, however, works differently. Instead of a method to directly process payments, it requires:

  1. Setting the customer ID.
  1. Entering the amount.
  1. Calling a makePayment() method to finalize the transaction.
// Legacy Payment System
class LegacyPaymentSystem {
    public void setCustomerId(String customerId) {
        System.out.println("Customer ID set to " + customerId);
    }

    public void setPaymentAmount(double amount) {
        System.out.println("Payment amount set to " + amount);
    }

    public void makePayment() {
        System.out.println("Payment processed successfully!");
    }
}

Issues Without Adapter Pattern:

  1. Incompatible Interfaces:
    • The modern payment system expects a method processPayment(String, double) to handle everything in one step.
    • The legacy system requires multiple steps (setCustomerId, setPaymentAmount, and makePayment), which doesn’t match the modern interface.
  1. No Direct Way to Reuse the Legacy System:
    • Without changing the legacy system itself (which may not be possible if it’s an external or third-party library), there’s no straightforward way to make it compatible with the modern interface.
  1. Maintenance Challenges:
    • Even if you try to write custom logic everywhere to handle the legacy system, you’ll end up duplicating code and increasing complexity, making the system harder to maintain.

Solution With Adapter Pattern

To solve this, we can use the Adapter Pattern. The idea is to create an adapter class that wraps the LegacyPaymentSystem and implements the ModernPaymentGateway interface. This adapter will internally translate the modern method calls into the legacy system’s calls.

// Adapter for Legacy Payment System
class LegacyPaymentAdapter implements ModernPaymentGateway {
    private LegacyPaymentSystem legacyPaymentSystem;

    public LegacyPaymentAdapter(LegacyPaymentSystem legacyPaymentSystem) {
        this.legacyPaymentSystem = legacyPaymentSystem;
    }

    @Override
    public void processPayment(String accountNumber, double amount) {
        // Translate the modern interface call to legacy system calls
        legacyPaymentSystem.setCustomerId(accountNumber);
        legacyPaymentSystem.setPaymentAmount(amount);
        legacyPaymentSystem.makePayment();
    }
}

Client Code

Now, the client code only interacts with the modern interface, and the adapter handles the complexity of using the legacy system:

public class ECommerceApplication {
    public static void main(String[] args) {
        // Using the modern payment gateway directly
        ModernPaymentGateway modernGateway = new SomeModernPaymentGateway();
        modernGateway.processPayment("123456", 500.0);

        // Using the legacy payment system through the adapter
        LegacyPaymentSystem legacySystem = new LegacyPaymentSystem();
        ModernPaymentGateway legacyAdapter = new LegacyPaymentAdapter(legacySystem);
        legacyAdapter.processPayment("987654", 1000.0);
    }
}

Benefits of Using the Adapter Pattern

  1. Reusing Legacy Code Without Modification:
    • The legacy system remains unchanged. The adapter wraps it and makes it compatible with the modern interface. This is crucial when working with third-party or external systems where you cannot modify the code.
  1. Uniform Interface:
    • The client code interacts with a single interface (ModernPaymentGateway) regardless of whether the underlying implementation is modern or legacy. This makes the client code simpler and easier to maintain.
  1. Easy Integration of New Systems:
    • If you need to integrate more legacy systems or switch to a new modern payment gateway, you only need to implement or update adapters without affecting the client code.
  1. Scalable and Maintainable Code:
    • The adapter encapsulates all the complexity of the legacy system. This reduces code duplication and makes the system easier to extend and maintain.

Conclusion

This example shows that the Adapter Pattern is essential when:

  • You need to integrate systems with incompatible interfaces.
  • Modifying existing systems is not an option.
  • You want to provide a unified interface to client code.

By using the Adapter Pattern, you achieve loose coupling, reusability, and scalability, making your codebase more maintainable and flexible for future changes.

Decorator Design Pattern Notes

Definition

  • Structural design pattern that allows dynamic addition of responsibilities to objects by wrapping them in decorator classes.
  • Alternative to subclassing; provides flexible extension of functionality at runtime.

Intent

  • Attach additional responsibilities to an object dynamically without affecting other instances of the same class.
  • Follows the Open/Closed Principle (open for extension, closed for modification).

Structure

        +----------------+          +------------------+
        |   Component    |          |    Decorator     |
        +----------------+          +------------------+
        | + operation()  |<---------| - component: Comp|
        +----------------+          | + operation()    |
                ^                   +------------------+
                |                             ^
                |                             |
+----------------------+             +------------------+
| ConcreteComponent    |             | ConcreteDecorator|
+----------------------+             +------------------+
| + operation()        |             | + operation()    |
+----------------------+             +------------------+

Participants

  1. Component
    • Interface/abstract class defining the operations (e.g., Coffee).
  1. ConcreteComponent
    • Implements the Component interface (e.g., SimpleCoffee).
  1. Decorator
    • Maintains a reference to a Component and implements the Component interface.
    • Base class for all decorators (e.g., CoffeeDecorator).
  1. ConcreteDecorator
    • Adds specific responsibilities (e.g., MilkDecorator, SugarDecorator).

Example Code

// Component
interface Coffee {
    double getCost();
    String getDescription();
}

// ConcreteComponent
class SimpleCoffee implements Coffee {
    public double getCost() { return 2.0; }
    public String getDescription() { return "Simple Coffee"; }
}

// Base Decorator
abstract class CoffeeDecorator implements Coffee {
    protected Coffee decoratedCoffee;
    public CoffeeDecorator(Coffee coffee) { this.decoratedCoffee = coffee; }
}

// ConcreteDecorator
class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee coffee) { super(coffee); }
    public double getCost() { return decoratedCoffee.getCost() + 0.5; }
    public String getDescription() {
        return decoratedCoffee.getDescription() + ", Milk";
    }
}

Usage:

Coffee coffee = new MilkDecorator(new SugarDecorator(new SimpleCoffee()));
System.out.println(coffee.getDescription()); // "Simple Coffee, Sugar, Milk"

Pros & Cons

✅ Pros

  • Flexibility: Add/remove responsibilities at runtime.
  • Avoids class explosion compared to subclassing.
  • Single Responsibility Principle: Each decorator focuses on one feature.

❌ Cons

  • Complexity: Many small classes may clutter the codebase.
  • Order-sensitive: Incorrect decorator order may lead to unintended behavior.
  • Hard to debug: Wrapping layers can complicate tracking.

Use Cases

  • Adding features to GUI components (e.g., scrollbars, borders).
  • Extending I/O functionality (e.g., Java BufferedInputStream).
  • Dynamic addition of logging, encryption, or compression.

Related Patterns

  • Adapter: Changes an object’s interface; Decorator enhances it.
  • Composite: Manages hierarchies, while Decorator adds responsibilities.
  • Strategy: Changes object’s behavior internally; Decorator changes it externally.

Implementation Steps

  1. Define the Component interface.
  1. Create ConcreteComponent classes.
  1. Build a base Decorator class that references the Component.
  1. Implement ConcreteDecorator classes to add functionalities.
  1. Compose decorators in client code as needed.

Real-World Example: Java I/O classes (FileInputStream, BufferedInputStream, GZIPInputStream).
