Java Multithreading

üßµ What is Multithreading in Java?

Multithreading is a programming technique where multiple threads run concurrently within a single process. It allows a program to perform multiple tasks at the same time, improving efficiency and responsiveness, especially in scenarios involving I/O operations, UI updates, or parallel computations.

  • In Java, multithreading is supported through the java.lang.Thread class and the Runnable interface.
  • Each thread has its own call stack but shares the heap memory with other threads in the same process.

A thread is an independent sequential path of execution within a program

‚úÖ Advantages of Multithreading:

  • Better CPU utilization (no idle time)
  • Efficient I/O handling (non-blocking UI)
  • Parallel execution of tasks
  • Simplified modeling of real-world concurrent behavior

üßµ Main Thread, User Thread & Daemon Thread in Java

üü¢ Main Thread

  • When a Java program starts, the JVM automatically creates one thread: the main thread.
  • This thread executes the main() method and is responsible for starting other threads if needed.
public class Main {
    public static void main(String[] args) {
        System.out.println("Current Thread: " + Thread.currentThread().getName());
    }
}
// Output: Current Thread: main

üßë‚Äçüíª User Thread (Non-Daemon Thread)

  • Default type of thread when you create one using new Thread().
  • The JVM keeps running as long as at least one user thread is alive.
  • Used for tasks that need to complete fully before the application exits.
Thread userThread = new Thread(() -> {
    System.out.println("User thread running...");
});
userThread.start();

üëª Daemon Thread

  • A background thread that runs in support of user threads.
  • JVM exits automatically when only daemon threads remain.
  • Use cases: garbage collection, monitoring, background logging, etc.

üîπ Marking a thread as daemon:

Thread daemonThread = new Thread(() -> {
    while (true) {
        System.out.println("Daemon thread running...");
    }
});
daemonThread.setDaemon(true); // Must be called before start()
daemonThread.start();
‚ö†Ô∏è If setDaemon(true) is called after start(), it throws IllegalThreadStateException.

1Ô∏è‚É£ Creating a Thread and Providing the Code That Gets Executed

üî∏ Learning View:

In Java, a thread represents a unit of execution. To create a thread and specify the task it should perform, you have two main options:

‚úÖ a. Extend the Thread class:

class MyThread extends Thread {
    public void run() {
        System.out.println("Thread is running...");
    }
}

MyThread t1 = new MyThread();
t1.start(); // Calls run() internally via a new thread

‚úÖ b. Implement the Runnable interface (preferred):

class MyRunnable implements Runnable {
    public void run() {
        System.out.println("Runnable thread running...");
    }
}

Thread t2 = new Thread(new MyRunnable());
t2.start();

‚úÖ c. Using Lambda (concise form from Java 8):

Thread t3 = new Thread(() -> System.out.println("Thread with lambda"));
t3.start();
üîç Why prefer Runnable?

Allows the class to extend another class (Java supports only single inheritance) and separates the task (run) from the thread control (Thread).

Q: How do you create a thread in Java? What's the difference between extending Thread and implementing Runnable?

Using the Runnable approach promotes loose coupling and better reusability.

2Ô∏è‚É£ Accessing Common Data ‚Äì The Role of Synchronization

üî∏ Learning View:

When multiple threads operate on shared data, inconsistencies may occur due to race conditions. To handle this, Java provides synchronization mechanisms.

‚úÖ Synchronizing a Method:

    count++;
}

‚úÖ Synchronizing a Block:

    count++;
}

üîπ Other ways to manage concurrency:

  • ReentrantLock ‚Äì more flexible than synchronized
  • AtomicInteger, ConcurrentHashMap ‚Äì thread-safe data structures from java.util.concurrent
üß† Always synchronize on shared resources, not local variables.

üíº Interview View:

Q: What is a race condition? How do you avoid it in Java?

A:

A race condition occurs when multiple threads access and modify shared data concurrently, leading to unpredictable results.

We avoid it using:

  • synchronized blocks/methods
  • Higher-level concurrency utilities (ReentrantLock, Atomic* classes)

Q: Difference between synchronized method and block?

A:

  • Method: locks the entire object
  • Block: allows more fine-grained control (e.g., synchronize only critical sections)

üßµ Creating a Thread from Runnable & Using Lambda ‚Äì Q&A

Q1: How do you create a thread using the Runnable interface?

A:

You implement the Runnable interface (which has a single run() method), then pass the instance to a Thread object and start it.

Example:

class MyTask implements Runnable {
    public void run() {
        System.out.println("Task is running...");
    }
}

Runnable task = new MyTask();
Thread t = new Thread(task);
t.start();

Q2: Why use Runnable instead of extending Thread?

A:

  • Java supports single inheritance ‚Üí implementing Runnable allows your class to extend another class.
  • Separates the task logic from the thread control, improving design and testability.
  • Encourages loose coupling between code and thread management.

Q3: How can we create a Runnable using a lambda function?

A:

Since Runnable is a functional interface (only one abstract method), we can use a lambda expression to provide its implementation.

Example:

Thread t = new Thread(() -> {
    System.out.println("Task via lambda is running...");
});
t.start();

Q4: How do we name the thread when using a lambda?

A:

You can use the two-argument Thread constructor:

Thread t = new Thread(() -> {
    System.out.println("Running: " + Thread.currentThread().getName());
}, "Worker-1");

t.start();

Output:

Running: Worker-1

Q5: What are the advantages of using a lambda for Runnable?

A:

  • More concise code (no separate class needed)
  • Improves readability for short tasks
  • Works well with streams & functional programming style

Q6: Common Interview Follow-Up

Q: Is a lambda-based Runnable any different from an anonymous inner class Runnable?

A: Functionally, they behave the same. The difference is syntax and readability ‚Äî lambdas are shorter and capture variables more cleanly.

‚ö° Race Condition & Synchronization in Java

üß© Race Condition

Definition

A race condition occurs when two or more threads try to access and modify shared data at the same time, and the final outcome depends on the timing of their execution.

Example (without synchronization):

class Counter {
    private int count = 0;
    public void increment() {
        count++;
    }
    public int getCount() {
        return count;
    }
}

Counter counter = new Counter();

Runnable task = () -> {
    for (int i = 0; i < 1000; i++) counter.increment();
};

Thread t1 = new Thread(task);
Thread t2 = new Thread(task);
t1.start();
t2.start();
t1.join();
t2.join();

System.out.println(counter.getCount()); // Expected: 2000, may be less due to race condition

Why less than 2000?

  • count++ is not atomic; it involves read ‚Üí increment ‚Üí write steps.
  • If threads interleave at the wrong time, increments are lost.

üõ° Synchronization

Definition

Synchronization ensures that only one thread can access a shared resource at a time, preventing race conditions.

1. Synchronized Method

public synchronized void increment() {
    count++;
}
  • Locks the entire method for the object (this).
  • Only one thread can execute this method at a time on the same object.

2. Synchronized Block

public void increment() {
    synchronized (this) {
        count++;
    }
}
  • Locks only the critical section.
  • Allows better performance by not locking unrelated code.

3. Static Synchronization

public static synchronized void incrementStatic() { ... }
  • Locks on the Class object (shared among all instances).

4. Advanced Alternatives

  • ReentrantLock ‚Äî More control over locking/unlocking.
  • AtomicInteger ‚Äî Performs atomic operations without explicit synchronization.
  • Concurrent collections like ConcurrentHashMap.

üîç Important Notes

  • Synchronization can reduce performance due to blocking.
  • Avoid synchronizing long operations or I/O to prevent bottlenecks.
  • Always synchronize on the smallest possible scope.

üíº Interview Q&A

Q1: What is a race condition?

A: A race condition occurs when multiple threads modify shared data concurrently without proper synchronization, causing unpredictable results.

Q2: How do you prevent race conditions in Java?

A: By using synchronization mechanisms such as:

  • synchronized methods or blocks
  • Lock API (ReentrantLock)
  • Atomic variables (AtomicInteger, etc.)
  • Thread-safe collections (ConcurrentHashMap)

Q3: Difference between synchronized method and synchronized block?

A:

  • Method: Locks the whole method.
  • Block: Locks only the critical section, improving performance.

Q4: Does synchronized make a method atomic?

A: Yes, for the scope of that method/block ‚Äî it ensures only one thread executes it at a time for the same lock object.

Q5: Can a synchronized method call another synchronized method?

A: Yes, Java locks are reentrant, so a thread can acquire the same lock multiple times without deadlock.

üì¶ Producer‚ÄìConsumer Problem

Definition

The Producer‚ÄìConsumer Problem is a classic multi-threading synchronization problem where:

  • Producer thread(s) generate data (produce items) and put them into a shared buffer.
  • Consumer thread(s) take data from the buffer and process it.
  • The buffer has a fixed size ‚Üí producers must wait if it‚Äôs full, consumers must wait if it‚Äôs empty.

Key Challenges

  1. Data consistency ‚Äì Multiple threads access the same buffer.
  1. Synchronization ‚Äì Avoid race conditions when adding/removing items.
  1. Coordination ‚Äì Ensure producers wait when buffer is full and consumers wait when buffer is empty.

Common Solutions

1. Using wait() and notify()

import java.util.LinkedList;
import java.util.Queue;

class Buffer {
    private Queue<Integer> queue = new LinkedList<>();
    private final int LIMIT = 5;

    public synchronized void produce(int value) throws InterruptedException {
        while (queue.size() == LIMIT) {
            wait(); // wait if buffer is full
        }
        queue.add(value);
        System.out.println("Produced: " + value);
        notifyAll(); // wake up consumers
    }

    public synchronized int consume() throws InterruptedException {
        while (queue.isEmpty()) {
            wait(); // wait if buffer is empty
        }
        int value = queue.poll();
        System.out.println("Consumed: " + value);
        notifyAll(); // wake up producers
        return value;
    }
}

public class ProducerConsumerExample {
    public static void main(String[] args) {
        Buffer buffer = new Buffer();

        Thread producer = new Thread(() -> {
            int value = 0;
            try {
                while (true) buffer.produce(value++);
            } catch (InterruptedException e) { e.printStackTrace(); }
        });

        Thread consumer = new Thread(() -> {
            try {
                while (true) buffer.consume();
            } catch (InterruptedException e) { e.printStackTrace(); }
        });

        producer.start();
        consumer.start();
    }
}

2. Using BlockingQueue (Simpler)

import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;

public class ProducerConsumerBQ {
    public static void main(String[] args) {
        BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(5);

        Thread producer = new Thread(() -> {
            int value = 0;
            try {
                while (true) {
                    queue.put(value);
                    System.out.println("Produced: " + value++);
                }
            } catch (InterruptedException e) { e.printStackTrace(); }
        });

        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    System.out.println("Consumed: " + queue.take());
                }
            } catch (InterruptedException e) { e.printStackTrace(); }
        });

        producer.start();
        consumer.start();
    }
}
  • put() waits if full
  • take() waits if empty
  • Automatically handles synchronization.

Interview Q&A

Q1: Why is synchronization needed in the Producer‚ÄìConsumer problem?

A: To prevent race conditions and ensure proper coordination between producer and consumer threads.

Q2: Difference between wait() and sleep() in this problem?

A:

  • wait() releases the lock and allows other threads to acquire it.
  • sleep() just pauses the thread but holds the lock.

Q3: Why use BlockingQueue over wait()/notify()?

A: It simplifies implementation by handling synchronization, blocking, and thread-safety internally.

Q4: What happens if you remove synchronization?

A: Race conditions may occur, items may be lost, or buffer may overflow/underflow.

üßµ Thread yield(), join(), and Scheduling in Java

1. Thread.yield()

Definition

yield() is a static method that hints to the thread scheduler:

"I‚Äôm willing to pause and let other threads run."

It does not guarantee that the thread will stop immediately or that other threads will run ‚Äî it‚Äôs just a request.

Key Points

  • Belongs to the current thread.
  • Gives equal-priority threads a chance to execute.
  • No effect on higher/lower priority threads if scheduler decides otherwise.
  • Used for fine-tuning CPU utilization, but rarely needed in modern Java.

Example:

Thread t1 = new Thread(() -> {
    for (int i = 0; i < 5; i++) {
        System.out.println(Thread.currentThread().getName());
        Thread.yield();
    }
});
t1.start();

2. join()

Definition

join() allows one thread to wait for the completion of another thread.

Key Points

  • Calling t.join() makes the current thread wait until thread t finishes.
  • Useful when you want sequential execution in a multi-threaded program.
  • Overloaded versions:
    • join() ‚Üí Wait indefinitely.
    • join(milliseconds) ‚Üí Wait up to given time.

Example:

Thread t1 = new Thread(() -> {
    for (int i = 0; i < 3; i++) System.out.println("Thread 1: " + i);
});

Thread t2 = new Thread(() -> {
    try {
        t1.join(); // t2 waits for t1 to finish
        System.out.println("Thread 2 starts after Thread 1");
    } catch (InterruptedException e) { e.printStackTrace(); }
});

t1.start();
t2.start();

3. Thread Scheduling

Definition

Thread scheduling is how the JVM + OS decide which thread runs at a given time.

Key Points

  • Managed by JVM + OS, not directly by you.
  • Based on:
    • Thread priority (1 to 10 in Java, default = 5)
    • OS-level scheduling algorithm (usually time slicing or preemptive priority)
  • Methods:
    • setPriority(int) ‚Äî Suggest priority (not guaranteed).
    • getPriority() ‚Äî Retrieve thread‚Äôs priority.

Interview Q&A

Q1: What‚Äôs the difference between sleep() and yield()?

A:

  • sleep() pauses the thread for a set time and holds CPU if uninterruptible.
  • yield() just hints the scheduler to switch to another thread.

Q2: Does yield() guarantee another thread will run?

A: No, it‚Äôs a hint ‚Äî the scheduler may ignore it.

Q3: What happens if you call join() on the current thread?

A: It will cause the thread to wait for itself ‚Üí deadlock.

Q4: Can thread priority ensure order of execution?

A: No, priority is just a suggestion to the scheduler; behavior is platform-dependent.

üéØ Thread Priority in Java

Definition

Thread priority is a suggestion to the thread scheduler about the importance of a thread compared to others.

In Java, each thread has a priority integer from 1 to 10.

Priority Constants

Java‚Äôs Thread class defines three commonly used constants:

ConstantValueMeaning
Thread.MIN_PRIORITY1Least important
Thread.NORM_PRIORITY5Default priority
Thread.MAX_PRIORITY10Most important

How It Works

  • Scheduler may prefer higher-priority threads, but priority is not a guarantee ‚Äî behavior is platform + JVM dependent.
  • Default priority = parent thread‚Äôs priority (main thread usually has priority 5).
  • Can be changed using:
thread.setPriority(int newPriority);
thread.getPriority();

Example

class PriorityExample {
    public static void main(String[] args) {
        Thread t1 = new Thread(() -> System.out.println("Low priority"));
        Thread t2 = new Thread(() -> System.out.println("High priority"));

        t1.setPriority(Thread.MIN_PRIORITY); // 1
        t2.setPriority(Thread.MAX_PRIORITY); // 10

        t1.start();
        t2.start();
    }
}

Note: Even with high priority, t2 might not run first.

Key Points

  • Thread priority affects scheduling, not execution speed.
  • Priority only works within the same JVM process.
  • OS scheduler ultimately decides execution order.
  • Overusing priorities can make programs unpredictable.

Interview Q&A

Q1: What‚Äôs the range of thread priority in Java?

A: 1 (min) to 10 (max), with default = 5.

Q2: Does higher priority mean faster execution?

A: No, it only increases the chance of being chosen first by the scheduler.

Q3: What happens if you set a priority outside 1‚Äì10?

A: IllegalArgumentException is thrown.

Q4: Can thread priority behavior vary across platforms?

A: Yes ‚Äî it depends on the OS thread scheduler‚Äôs implementation.
