The Open/Closed Principle (OCP) is one of the five SOLID principles of object-oriented design. It states that:

  • Software entities (such as classes, modules, functions, etc.) should be open for extension but closed for modification.

What does this mean?

  • Open for extension: You should be able to extend or add new functionality to a class or module without changing its existing code.
  • Closed for modification: Once a class or module is developed and tested, its source code should not be modified unless necessary. Instead, new behavior should be added by extending the class or module.

In this scenario, different interview questions require specific actions, but this design still violates OCP because any time a new question type is introduced, we need to modify the InterviewQuestionProcessor.

Violation of OCP :

// Interface for interview questions
interface InterviewQuestion {
    void askQuestion();
}

// Concrete classes
class MathInterviewQuestion implements InterviewQuestion {
    @Override
    public void askQuestion() {
        System.out.println("Solve: What is 10 * 10?");
    }
}

class ScienceInterviewQuestion implements InterviewQuestion {
    @Override
    public void askQuestion() {
        System.out.println("Explain the process of photosynthesis.");
    }
}

// Processor that violates OCP by adding custom logic for each question type
class InterviewQuestionProcessor {
    public void processQuestion(InterviewQuestion question) {
        if (question instanceof MathInterviewQuestion) {
            System.out.println("Showing math question: ");
            question.askQuestion();
            System.out.println("Provide a calculator for assistance.");
        } else if (question instanceof ScienceInterviewQuestion) {
            System.out.println("Showing science question: ");
            question.askQuestion();
            System.out.println("Provide reference material for diagrams.");
        }
    }
}

Why This Violates OCP:

  • Different logic: The if block for MathInterviewQuestion adds extra logic (providing a calculator), while the else if block for ScienceInterviewQuestion adds different logic (providing reference material).
  • OCP Violation: Every time a new question type (e.g., HistoryInterviewQuestion) is added, you'll need to modify the InterviewQuestionProcessor and add more custom logic in a new if or else if block, which violates the Open/Closed Principle.

Fix (Adhering to OCP):

In this fix, we delegate the specific behavior (e.g., showing a calculator or reference material) to each question class, removing the need to modify InterviewQuestionProcessor when adding new question types.

// Interface for interview questions
interface InterviewQuestion {
    void askQuestion();
    void provideHelp();  // New method to handle different logic
}

// Concrete classes
class MathInterviewQuestion implements InterviewQuestion {
    @Override
    public void askQuestion() {
        System.out.println("Solve: What is 10 * 10?");
    }

    @Override
    public void provideHelp() {
        System.out.println("Provide a calculator for assistance.");
    }
}

class ScienceInterviewQuestion implements InterviewQuestion {
    @Override
    public void askQuestion() {
        System.out.println("Explain the process of photosynthesis.");
    }

    @Override
    public void provideHelp() {
        System.out.println("Provide reference material for diagrams.");
    }
}

// New question type without modifying processor
class HistoryInterviewQuestion implements InterviewQuestion {
    @Override
    public void askQuestion() {
        System.out.println("Who was the first president of the USA?");
    }

    @Override
    public void provideHelp() {
        System.out.println("Provide historical timelines.");
    }
}

// Processor that adheres to OCP
class InterviewQuestionProcessor {
    public void processQuestion(InterviewQuestion question) {
        System.out.println("Showing question: ");
        question.askQuestion();
        question.provideHelp();  // Now uses polymorphism
    }
}

Key Idea:

  • Violation: The first version violates OCP because the processor contains specific logic for each question type, and adding new types requires modifying the processor.
  • Fix: The second version adheres to OCP by pushing the specific logic (like providing help materials) into the individual question classes. Now, the InterviewQuestionProcessor uses polymorphism and never needs to be modified when new question types are added.

The Interface Segregation Principle (ISP) in SOLID states that a class should not be forced to implement interfaces it does not use. In other words, an interface should have only the methods that are relevant to the specific behavior it represents.

Example Without Following ISP

// A single, large interface
interface Worker {
    void work();           // All workers can work
    void eat();            // All workers must eat
    void attendMeetings(); // All workers must attend meetings
}

// Low-level modules
class Developer implements Worker {
    @Override
    public void work() {
        System.out.println("Developer writing code...");
    }

    @Override
    public void eat() {
        System.out.println("Developer having lunch...");
    }

    @Override
    public void attendMeetings() {
        System.out.println("Developer attending technical meetings...");
    }
}

class Robot implements Worker {
    @Override
    public void work() {
        System.out.println("Robot performing automated tasks...");
    }

    @Override
    public void eat() {
        // Robots don't eat, but they are forced to implement this method
        throw new UnsupportedOperationException("Robots do not eat!");
    }

    @Override
    public void attendMeetings() {
        // Robots don't attend meetings, but they are forced to implement this method
        throw new UnsupportedOperationException("Robots do not attend meetings!");
    }
}

Issues Without ISP:

  1. Unnecessary Implementation: Robot is forced to implement eat() and attendMeetings(), which are irrelevant for robots.
  1. Violation of Interface Segregation: Clients using the Worker interface (e.g., Robot users) have to deal with methods they do not need, leading to potential misuse or runtime exceptions.
  1. Rigid Design: Adding new types of workers with unique requirements might require frequent changes to the large interface, violating the Open/Closed Principle.

Example Following ISP

Refactor the interface into smaller, specific interfaces, each representing a distinct responsibility:

// Small, specific interfaces
interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

interface MeetingAttender {
    void attendMeetings();
}

// Implementations
class Developer implements Workable, Eatable, MeetingAttender {
    @Override
    public void work() {
        System.out.println("Developer writing code...");
    }

    @Override
    public void eat() {
        System.out.println("Developer having lunch...");
    }

    @Override
    public void attendMeetings() {
        System.out.println("Developer attending technical meetings...");
    }
}

class Robot implements Workable {
    @Override
    public void work() {
        System.out.println("Robot performing automated tasks...");
    }
}

Benefits of ISP:

  1. Clearer Design: Each interface has a specific responsibility, making it easier to understand and use.
  1. Flexible Code: New types of workers can implement only the interfaces relevant to them.
  1. Avoid Unnecessary Methods: Robot now implements only the Workable interface, avoiding the need to implement irrelevant methods like eat() or attendMeetings().

Usage in a Client Class

public class WorkStation {
    public void startWork(Workable worker) {
        worker.work();
    }

    public void lunchBreak(Eatable worker) {
        worker.eat();
    }

    public void scheduleMeeting(MeetingAttender worker) {
        worker.attendMeetings();
    }

    public static void main(String[] args) {
        WorkStation station = new WorkStation();

        Developer developer = new Developer();
        station.startWork(developer);   // Developer working
        station.lunchBreak(developer); // Developer eating
        station.scheduleMeeting(developer); // Developer attending meetings

        Robot robot = new Robot();
        station.startWork(robot);      // Robot working
        // station.lunchBreak(robot); // Compilation error! Robots cannot eat
        // station.scheduleMeeting(robot); // Compilation error! Robots cannot attend meetings
    }
}

Key Takeaways:

  • Without ISP: Classes are forced to implement irrelevant methods, leading to potential misuse and brittle designs.
  • With ISP: Classes implement only the behaviors they require, resulting in a cleaner, more modular design.
  • Practical Resolution: Always break large interfaces into smaller, specific ones that focus on distinct responsibilities.

The D in SOLID stands for the Dependency Inversion Principle (DIP). It suggests that high-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details; details should depend on abstractions.

Code Example Without Following DIP

// Low-level module
class Keyboard {
    public void type() {
        System.out.println("Typing on Keyboard...");
    }
}

// High-level module
class Computer {
    private Keyboard keyboard;

    public Computer() {
        // Direct dependency on a concrete implementation
        this.keyboard = new Keyboard();
    }

    public void useComputer() {
        keyboard.type();
    }
}

public class Main {
    public static void main(String[] args) {
        Computer computer = new Computer();
        computer.useComputer();
    }
}

Issues with this Approach:

  1. Tightly Coupled Code: The Computer class depends on the concrete Keyboard class. If we need to use a different type of input device, like a VirtualKeyboard, we have to modify the Computer class.
  1. Poor Scalability: Adding new input devices requires changing the Computer class, violating the Open/Closed Principle.
  1. Difficult Testing: It’s hard to mock or replace the Keyboard class for unit testing.

Code Example Following DIP

Here’s how we can refactor the code to follow the Dependency Inversion Principle:

// Abstraction
interface InputDevice {
    void input();
}

// Low-level module
class Keyboard implements InputDevice {
    @Override
    public void input() {
        System.out.println("Typing on Keyboard...");
    }
}

// Another Low-level module
class VirtualKeyboard implements InputDevice {
    @Override
    public void input() {
        System.out.println("Typing on Virtual Keyboard...");
    }
}

// High-level module
class Computer {
    private InputDevice inputDevice;

    // Dependency injected via constructor
    public Computer(InputDevice inputDevice) {
        this.inputDevice = inputDevice;
    }

    public void useComputer() {
        inputDevice.input();
    }
}

public class Main {
    public static void main(String[] args) {
        // Inject Keyboard as the dependency
        InputDevice keyboard = new Keyboard();
        Computer computer = new Computer(keyboard);
        computer.useComputer();

        // Inject VirtualKeyboard as the dependency
        InputDevice virtualKeyboard = new VirtualKeyboard();
        Computer computerWithVirtualKeyboard = new Computer(virtualKeyboard);
        computerWithVirtualKeyboard.useComputer();
    }
}

How DIP Resolves the Issue:

  1. Loosely Coupled Code: The Computer class depends on the abstraction InputDevice, not on a specific implementation like Keyboard or VirtualKeyboard.
  1. Better Scalability: Adding new input devices (e.g., a TouchScreen) doesn’t require changing the Computer class.
  1. Improved Testability: You can easily mock the InputDevice interface for unit testing without modifying the Computer class.

By adhering to DIP, your code becomes more maintainable, extensible, and testable.
