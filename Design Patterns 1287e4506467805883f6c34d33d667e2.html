package com.globalsoftwaresupport;

public class DatabaseConnector {

    // Lazy initialization of the singleton instance.
    // The instance is not created until the first time it is needed.
    private static DatabaseConnector INSTANCE;

    // Private constructor to prevent external instantiation.
    // This ensures that the class cannot be instantiated from outside this class.
    private DatabaseConnector() {
        // Prevent reflection attack
        // We can add extra protection to prevent instantiation via reflection,
        // but that's a more advanced step.
    }

    // Public method to provide global access to the singleton instance.
    // This method ensures only one instance of the class is created.
    public static DatabaseConnector getInstance() {

        // Check if the instance is null (i.e., has not been created yet).
        if(INSTANCE == null) {
            // If multiple threads access this method simultaneously, they might both
            // try to create the instance.Using synchronized ensures that only one thread
            // can enter this block and create the instance.
            synchronized(DatabaseConnector.class) {
                // If INSTANCE is still null, create a new instance.
                // This block guarantees that the instance is created only once.
                if (INSTANCE == null) {
                    INSTANCE = new DatabaseConnector();
                }
            }
        }

        // Return the singleton instance.
        return INSTANCE;
    }

    // Method to simulate connecting to a database.
    public void connect() {
        System.out.println("Connecting to the database...");
    }

    // Method to simulate disconnecting from a database.
    public void disconnect() {
        System.out.println("Disconnecting...");
    }
}

Notes:

  • Lazy Initialization: The instance is only created when getInstance() is called for the first time, optimizing resource usage.
  • Thread Safety: The synchronized block ensures that only one thread can create the instance at a time, preventing race conditions in a multithreaded environment.
  • Double-Checked Locking: Inside the synchronized block, the instance is checked again (if (INSTANCE == null)) to prevent multiple threads from creating instances.
  • Private Constructor: Ensures the class cannot be instantiated from outside, making it a true singleton.

üè≠ Factory Design Pattern ‚Äì Notes

üìå Definition

The Factory Pattern provides an interface to create objects but allows subclasses or a factory method to decide which class to instantiate.

üîß Use when: The object creation logic depends on input or varies by type, and you want to encapsulate this logic outside of your core business code.

üîç Why Use It?

  • ‚úÖ Avoids tight coupling between object creation and usage
  • ‚úÖ Promotes Open/Closed Principle ‚Äì add new types without modifying core code
  • ‚úÖ Makes the code more testable, extensible, and modular
  • ‚úÖ Ideal when you have a common interface and multiple implementations

üõ†Ô∏è How to Identify When to Use Factory Pattern

Ask yourself:

‚ÄúDo I have multiple types of an object that share behavior but differ in implementation?‚Äù

If yes, Factory Pattern is a good fit.

üß± Structure

// Step 1: Define the interface
interface Notification {
    void send();
}

// Step 2: Create subclasses
class SMSNotification implements Notification {
    public void send() {
        System.out.println("Sending SMS");
    }
}

class EmailNotification implements Notification {
    public void send() {
        System.out.println("Sending Email");
    }
}

// Step 3: Create the Factory
class NotificationFactory {
    public static Notification getNotification(String type) {
        if (type.equalsIgnoreCase("sms")) return new SMSNotification();
        else if (type.equalsIgnoreCase("email")) return new EmailNotification();
        throw new IllegalArgumentException("Unknown type");
    }
}

// Step 4: Client code
Notification notification = NotificationFactory.getNotification("sms");
notification.send();

üß† Key Design Principles

  • Open/Closed Principle (OCP): Add new types without modifying existing code.
  • Single Responsibility Principle (SRP): Factory class handles object creation.
  • Dependency Inversion: High-level modules depend on abstractions, not concrete classes.

‚ö†Ô∏è Common Mistakes to Avoid

  • ‚ùå Mixing creation logic inside the main class (violates SRP & OCP)
  • ‚ùå Not handling invalid input properly
  • ‚ùå Forgetting to define a clear abstraction (interface or base class)

üßµ One-Liner Summary

"Use the Factory Pattern when your code needs to create objects based on conditions without exposing the instantiation logic."

üëÅÔ∏è‚Äçüó®Ô∏è Observer Design Pattern ‚Äì Notes

üìå Definition

The Observer Pattern defines a one-to-many dependency between objects so that when one object (the Subject) changes its state, all its dependents (Observers) are automatically notified and updated.

üîç When to Use

  • When multiple components need to react to changes in another object.
  • You want to implement an event system without tightly coupling the source and listeners.
  • Especially useful in real-time update systems or publish/subscribe models.

üß† Real-world Analogy

Think of a YouTube Channel (Subject) and its Subscribers (Observers):

  • When a new video is uploaded (state change),
  • All subscribers are notified.

üõ†Ô∏è Structure

// Step 1: Observer Interface
interface Observer {
    void update(String message);
}

// Step 2: Subject Interface
interface Subject {
    void register(Observer o);
    void unregister(Observer o);
    void notifyObservers(String message);
}

// Step 3: Concrete Subject
class Channel implements Subject {
    private List<Observer> observers = new ArrayList<>();

    public void register(Observer o) { observers.add(o); }
    public void unregister(Observer o) { observers.remove(o); }

    public void notifyObservers(String message) {
        for (Observer o : observers) {
            o.update(message);
        }
    }

    public void uploadNewVideo(String videoTitle) {
        notifyObservers("New video uploaded: " + videoTitle);
    }
}

// Step 4: Concrete Observer
class Subscriber implements Observer {
    private String name;
    public Subscriber(String name) { this.name = name; }

    public void update(String message) {
        System.out.println(name + " received: " + message);
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        Channel ytChannel = new Channel();
        Observer user1 = new Subscriber("Lakshman");
        Observer user2 = new Subscriber("Ravi");

        ytChannel.register(user1);
        ytChannel.register(user2);

        ytChannel.uploadNewVideo("Observer Pattern in 5 Minutes!");
    }
}

üß† Strategy Design Pattern

üìå Purpose:

Define a family of algorithms, encapsulate each one, and make them interchangeable.

Used when you want to select an algorithm at runtime, decouple business logic from specific implementations.

üì¶ Real-world Analogy ‚Äì Payment Gateway

Let‚Äôs say you are building an e-commerce app. It supports:

  • ‚úÖ Multiple payment methods: Credit Card, PayPal, UPI.
  • ‚úÖ Multiple discount strategies: New User, Festival, No Discount.
  • ‚úÖ Multiple tax rules: Based on region (India, US, EU).

Each of these can be swapped independently, and they follow Strategy Pattern.

üéÆ Context Class

public class PaymentProcessor {
    private PaymentStrategy paymentStrategy;
    private DiscountStrategy discountStrategy;
    private TaxStrategy taxStrategy;

    public PaymentProcessor(PaymentStrategy p, DiscountStrategy d, TaxStrategy t) {
        this.paymentStrategy = p;
        this.discountStrategy = d;
        this.taxStrategy = t;
    }

    public void processPayment(double baseAmount) {
        double discounted = discountStrategy.applyDiscount(baseAmount);
        double tax = taxStrategy.calculateTax(discounted);
        double finalAmount = discounted + tax;
        paymentStrategy.pay(finalAmount);
    }
}

public class Main {

        TaxStrategy tax = new IndiaTax();

        PaymentProcessor processor = new PaymentProcessor(payment, discount, tax);
        processor.processPayment(1000.0);
    }
}

  • Cleaner, modular design
  • Can unit test each strategy independently
